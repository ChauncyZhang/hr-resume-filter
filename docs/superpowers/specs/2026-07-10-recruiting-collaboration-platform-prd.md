## 2. 产品摘要

现有项目是一个本地简历初筛工具：HR 保存岗位 JD，批量上传简历，系统按规则和可选 LLM 生成排序、匹配理由、风险点和建议面试问题，再导出 CSV。

目标产品要从“一次性文件筛选工具”升级为“招聘过程的系统记录和协作平台”。HR、用人经理和面试官围绕职位与候选人，在同一套网页系统中完成：

1. 创建和管理职位。
2. 导入、解析、去重和筛选简历。
3. 推进候选人招聘阶段。
4. 安排面试并收集结构化反馈。
5. 记录招聘结果和操作历史。
6. 将合适但暂未录用的候选人沉淀到人才库。
7. 在后续职位中搜索并重新激活历史人才。

首期采用单公司私有部署，面向中小规模招聘团队。产品借鉴成熟 ATS 的对象模型和工作流，但不在首期建设招聘渠道生态、校招、Offer 审批、入职或候选人营销能力。

## 3. 背景与现状

### 3.1 当前能力

当前仓库已经具备：

- 保存和复用岗位名称、JD。
- 批量上传 TXT、MD、CSV、PDF、DOCX 简历。
- PDF 文本层与 DOCX 文本解析。
- 按必须条件、加分项和经验年限进行规则评分。
- 可选 OpenAI-compatible 或 Ollama LLM 评分。
- 输出匹配理由、风险点和建议面试问题。
- 按文件显示筛选进度和结果卡片。
- CSV 导出。
- Windows、macOS 本地启动和 Linux 基础部署说明。

### 3.2 当前限制

当前产品不具备：

- 用户登录、角色、部门和对象级权限。
- 服务端数据库、候选人档案和长期文件存储。
- 候选人与职位申请的独立数据模型。
- 简历版本、重复候选人、来源和历史投递记录。
- 招聘流程状态机、负责人、待办和协作记录。
- 面试安排、面试官任务、反馈和结果汇总。
- 人才库、标签、检索和重新激活。
- 审计日志、数据保留、删除和导出治理。
- 企业级 LLM Provider 管理和数据外发控制。

本地 JSON、临时文件和一次性筛选结果适合个人工具，不适合多人招聘系统的共享、追溯和权限要求。

## 4. 问题陈述

招聘过程中的主要问题不是单纯“筛简历慢”，而是信息和责任分散：

- 简历来自多个来源，重复、版本和历史沟通难以识别。
- HR 初筛结果存在于文件或表格中，无法持续推进和追踪。
- 用人经理看不到统一、可解释的候选人信息。
- 面试安排依赖聊天和手工日历，变更与反馈容易遗漏。
- 面试官反馈格式不统一，HR 需要反复催促和汇总。
- 未录用但有价值的候选人没有结构化沉淀，后续职位无法复用。
- AI 评分缺少版本、权限和审计，容易被误用为自动决策。

产品需要建立一个共享、可追溯、有权限边界的招聘工作流，同时保留现有简历筛选能力带来的效率价值。

## 5. 产品目标

### 5.1 业务目标

- 将“简历导入到面试结果”纳入一个连续招聘流程。
- 将简历初筛时间相对当前人工基线降低 50%。
- 让 HR 随时看到每个职位和候选人的当前状态、负责人和下一步动作。
- 让面试官以低操作成本提交标准化反馈。
- 让历史候选人可以被检索、归档、授权访问和重新用于新职位。

### 5.2 产品目标

- 建立职位、候选人、职位申请、简历、筛选、面试、反馈和人才库的核心对象。
- 支持 HR、用人经理、面试官和管理员的角色协作。
- 将规则和 LLM 评分变成可解释、可版本化、可审计的辅助能力。
- 支持公司服务器部署、HTTPS 访问、备份恢复和基础可观测性。
- 保持现有合法简历来源边界，不自动登录或抓取第三方招聘平台。

### 5.3 非目标

首期不包含：

- 多公司、多租户 SaaS、租户开通和计费。
- 自动登录 BOSS 直聘或其他招聘平台、绕过验证码、抓取未授权简历。
- 多渠道职位自动发布和简历自动回流。
- 招聘官网、候选人门户、内推门户、猎头门户。
- 校招项目、叫号面试、批量考试或测评。
- Offer 审批、电子签、背调、入职和员工档案。
- 短信、邮件营销、自动外呼和复杂人才 CRM。
- AI 自动淘汰、自动录用或代替人工作出招聘决定。
- 原生移动 App；移动端仅提供必要查看和反馈能力。

## 6. 产品原则

1. **流程优先于功能堆叠**：每个功能必须服务于候选人的完整招聘流转。
2. **候选人与申请分离**：一个候选人可拥有多份简历、参与多个职位流程。
3. **人工决策优先**：规则和 LLM 只辅助排序与总结，不自动淘汰候选人。
4. **证据可解释**：评分必须展示命中项、缺失项、风险和所使用的版本。
5. **权限默认收紧**：面试官只看到完成任务所需的候选人信息。
6. **失败可降级**：单份简历或 LLM 失败不得阻断整批任务和规则评分。
7. **历史不可覆盖**：新简历、新职位和新面试不能覆盖旧的申请与反馈记录。
8. **服务器不等于可信**：内网部署仍必须登录、授权、审计和加密。

## 7. 用户与核心任务

### 7.1 HR / 招聘专员

核心任务：

- 创建职位、维护 JD 和筛选标准。
- 批量导入简历并查看筛选排序。
- 复核候选人、添加备注和推进招聘阶段。
- 安排面试、处理改期和催收反馈。
- 将候选人归档或加入人才库。

成功标准：不依赖外部 Excel 也能知道候选人当前状态和下一步动作。

### 7.2 招聘负责人

核心任务：

- 查看团队职位、候选人漏斗和待办积压。
- 分配职位负责人和监督流程完整性。
- 管理流程模板、人才库规则和导出权限。

成功标准：能够发现卡点，并追溯职位或候选人的历史操作。

### 7.3 用人经理

核心任务：

- 查看本人负责或被授权职位的候选人。
- 查看筛选依据、简历、HR 备注和面试结果。
- 给出推进、保留或淘汰建议。

成功标准：无需向 HR 反复索要材料即可完成业务判断。

### 7.4 面试官

核心任务：

- 查看本人被安排的面试和必要候选人资料。
- 查看职位重点和建议面试问题。
- 在面试后提交结构化评价和结论。

成功标准：面试材料集中，反馈提交不超过三分钟。

### 7.5 系统管理员

核心任务：

- 管理账号、角色、认证、集成、LLM 和系统配置。
- 查看安全与系统审计，不默认查看所有简历正文。
- 管理备份、数据保留和删除策略。

成功标准：能够控制数据访问和外发，不接触不必要的候选人内容。

## 8. 信息架构

一级导航：

1. **工作台**：今日待办、新简历、待安排面试、待反馈、待决策和异常任务。
2. **职位**：职位列表、职位详情、JD、筛选规则和候选人漏斗。
3. **候选人**：跨职位候选人搜索、筛选、标签和历史记录。
4. **面试**：面试列表、日历、待安排、待反馈和历史记录。
5. **人才库**：人才分组、标签、搜索、重新激活和保留期限。
6. **报表**：首期基础漏斗与效率指标；高级分析在后续阶段。
7. **设置**：账号权限、部门、流程、面试评价、LLM、审计和数据治理。

默认入口为工作台。简历批量筛选作为职位详情中的核心操作和可追溯任务，不单独成为孤立工具页。

页面级信息架构、角色旅程、交互状态和可用性验证见 [UX 设计规范](../../design/recruiting-platform-ux-spec.md)；视觉基础、布局、组件和响应式规则见 [UI 设计规范](../../design/recruiting-platform-ui-spec.md)。

## 9. 核心对象模型

### 9.1 关键对象

| 对象 | 说明 |
| --- | --- |
| 用户 User | 系统使用者，关联角色、部门和可见范围 |
| 职位 Job | 招聘岗位主对象，包含负责人、状态和招聘人数 |
| JD 版本 JobJDVersion | 职位描述和筛选标准版本 |
| 候选人 Candidate | 真实自然人档案，不绑定单一职位 |
| 联系方式 CandidateContact | 手机、邮箱等多值联系方式，支持脱敏和去重 |
| 简历 Resume | 候选人的某一份文件和解析结果 |
| 职位申请 Application | 候选人针对某职位的一次招聘流程 |
| 筛选任务 ScreeningRun | 某职位和一批简历的一次筛选任务 |
| 筛选结果 ScreeningResult | 规则与 LLM 对某次申请的评分和解释 |
| 面试 Interview | 某次申请的一轮面试安排 |
| 面试反馈 InterviewFeedback | 某面试官针对一轮面试提交的评价 |
| 人才库 TalentPool | 可配置的人才分组 |
| 人才库成员 TalentPoolEntry | 候选人与人才库的关联、标签和保留信息 |
| 审计事件 AuditLog | 查看、下载、导出、修改和配置变更记录 |

### 9.2 数据不变量

- 一个候选人可以拥有多份简历。
- 一个候选人可以拥有多个职位申请。
- 一个职位申请只属于一个职位。
- 一个职位申请可以拥有多次筛选结果和多轮面试。
- 筛选结果必须引用 JD 版本、简历版本、规则版本和模型版本。
- 人才库成员关系独立于职位申请状态；加入人才库不能覆盖历史申请。
- 合并候选人时保留原始对象、申请、简历和审计映射。

## 10. 状态模型

### 10.1 职位状态

```text
草稿 -> 招聘中 -> 暂停 -> 招聘中
招聘中/暂停 -> 已关闭 -> 已归档
```

### 10.2 职位申请状态

```text
新简历
-> 待复核
-> 待沟通
-> 待安排面试
-> 面试中
-> 待决策
-> 通过
-> 已录用
```

任一非终态可进入 `已淘汰` 或 `已撤回`。`已淘汰`、`已撤回` 和 `已录用` 为不可直接重开的终态。候选人再次参与同一或其他职位时创建新的职位申请，并通过 `来源申请` 字段关联原申请；原状态和历史记录保持不变。

解析状态、筛选状态和人才库状态不混入职位申请状态，避免一个状态字段同时表达多类业务事实。

### 10.3 筛选任务状态

```text
排队中 -> 解析中 -> 规则评分中 -> LLM评分中 -> 已完成
                                   -> 部分成功
任一执行态 -> 已失败/已取消
```

LLM 未启用时跳过 LLM 阶段。LLM 失败时保留规则评分，并将任务标记为部分成功。

### 10.4 面试状态

```text
草稿 -> 已安排 -> 已确认 -> 已完成 -> 待反馈 -> 反馈完成
已安排/已确认 -> 已改期 -> 已安排
已安排/已确认 -> 已取消
已确认 -> 候选人未到场
```

所有状态变化由服务端校验，记录操作者、前后状态、时间和原因。

## 11. 核心用户流程

### 11.1 创建职位

1. HR 输入职位名称、部门、招聘人数、负责人和 JD。
2. 系统从 JD 中提取必须条件和加分项，HR 可人工修改。
3. HR 选择是否启用该职位的 LLM 辅助评分。
4. 保存时生成 JD 版本和筛选规则版本。
5. 职位切换为招聘中后可导入简历。

### 11.2 导入并筛选简历

1. HR 在职位详情上传一批简历，并选择来源和上传批次说明。
2. 系统为每个文件创建处理记录，并异步解析。
3. 系统提取姓名、手机、邮箱、技能、教育和经历等可识别字段。
4. 系统根据文件哈希、邮箱、手机号等给出重复提示，不自动合并。
5. 规则评分完成后，按岗位设置决定是否调用 LLM。
6. 页面持续显示 `当前份数/总份数`，刷新后仍可恢复任务进度。
7. 完成后展示排序、命中项、缺失项、风险、建议问题和失败原因。
8. HR 可批量推进到待沟通、标记淘汰或加入人才库。

### 11.3 HR 与用人经理复核

1. HR 打开候选人详情，查看简历、结构化信息和筛选依据。
2. HR 添加备注、标签和人工结论。
3. 用人经理仅查看被授权职位的候选人并提交建议。
4. 人工结论覆盖展示优先级，但不删除原始规则和 LLM 结果。

### 11.4 安排面试

1. HR 从候选人详情或职位漏斗发起安排。
2. 选择轮次、面试方式、时间、时区、面试官和地点或会议链接。
3. 系统检查同一面试官和候选人的已知时间冲突。
4. 保存后生成面试官待办、标准邀请文本和 `.ics` 日历文件，并记录候选人通知状态。
5. 未配置通知集成时，HR 可复制邀请文本或下载 `.ics` 后通过现有沟通工具发送。
6. HR 可改期、取消、追加面试官和记录候选人未到场。

首期保证系统内安排与待办闭环。邮件、日历和即时通讯自动通知作为可插拔集成，未配置集成时不阻断本地面试记录。

### 11.5 面试反馈与决策

1. 面试官进入“待反馈”查看候选人摘要、JD 要点、简历和建议问题。
2. 面试官填写结构化维度、优点、风险、结论和建议。
3. 结论选项为：强烈推荐、推荐、保留、不推荐。
4. 普通面试官不能修改他人反馈。
5. 所有必需反馈完成后，申请进入待决策。
6. HR 或有权限的用人经理决定通过、淘汰、追加面试或加入人才库。

### 11.6 人才库沉淀与重新激活

1. HR 主动选择将候选人加入一个或多个人才库。
2. 记录适合职位、技能标签、跟进人、来源、加入原因和保留期限。
3. 人才库支持姓名、联系方式、技能、标签、城市、历史职位和最近互动检索。
4. 重新激活时为目标职位创建新的职位申请，保留原历史。
5. “暂不适合”“永久不再联系”和“黑名单”必须使用不同语义和权限。

## 12. 功能需求

### 12.1 登录、组织和权限（P0）

- 所有业务页面和 API 必须登录后访问。
- 支持管理员创建、禁用和重置本地账号。
- 支持接入 OIDC/SSO；未配置企业身份源时使用受控本地账号。
- 角色至少包括系统管理员、招聘管理员、HR、用人经理、面试官。
- 权限必须在服务端按角色、部门、职位负责人和面试分配进行校验。
- 系统管理员默认无权查看全部简历内容，除非同时具有招聘数据权限。
- 登录会话使用安全 Cookie、CSRF 防护、超时和退出失效机制。

### 12.2 工作台（P0）

- 展示本人负责职位的待处理数量。
- 展示新简历、待复核、待安排面试、今日面试、待反馈和待决策。
- 展示解析失败、LLM 部分失败、通知失败等异常任务。
- 支持从待办直接进入对应候选人或面试。
- 主操作为新建职位和导入简历。

### 12.3 职位管理（P0）

- 创建、编辑、暂停、关闭和归档职位。
- 管理负责人、部门、招聘人数、优先级和状态。
- 保存公开 JD 和内部筛选标准。
- JD 和筛选标准每次生效变更生成新版本。
- 职位详情展示候选人漏斗、筛选任务和最近活动。
- 支持列表和阶段看板两种候选人视图。

### 12.4 候选人和简历（P0）

- 创建候选人档案并允许人工修正解析字段。
- 保存原始简历、解析文本、文件元数据和解析质量。
- 支持同一候选人的多份简历和多次投递。
- 候选人详情展示完整时间线。
- 联系方式在无权限列表中默认脱敏。
- 提供重复候选人提示，合并必须人工确认并可追溯。
- 解析失败显示具体文件、阶段和可重试操作。

### 12.5 筛选任务（P0）

- 每次批量筛选形成持久化任务，不因页面关闭而丢失。
- 单文件失败不阻断同批其他文件。
- 支持按规则分、LLM 分、状态、关键词、来源和上传批次筛选。
- 规则与 LLM 结果分别展示，不用单一分数掩盖差异。
- HR 可记录人工覆盖结论和原因。
- 支持导出当前有权限的筛选结果。

### 12.6 招聘流程（P0）

- 支持申请状态流转、批量操作和状态原因。
- 非法状态跳转由服务端拒绝。
- 每次状态变化写入候选人时间线和审计日志。
- 并发修改使用版本校验，冲突时不静默覆盖。
- 淘汰必须选择或填写原因；撤回与淘汰分开统计。

### 12.7 面试管理（P0）

- 支持面试轮次、方式、地点、会议链接、面试官和时间。
- 支持列表和周日历视图。
- 支持改期、取消、未到场和补充面试官。
- 检查系统内已知时间冲突并提示。
- 生成标准邀请文本和 `.ics` 日历文件，支持 HR 通过现有沟通工具发送。
- 面试完成后自动生成待反馈任务。
- 通知和日历集成失败时保留本地安排，并明确显示失败状态。

### 12.8 面试反馈（P0）

- 面试官只看到被分配面试所需的候选人材料。
- 支持结构化评分、优点、风险、结论和补充说明。
- 反馈草稿仅本人可见；提交后按权限向 HR 和用人经理展示。
- 普通面试官不能修改他人反馈。
- 修改已提交反馈必须记录修改前后内容和理由。

### 12.9 人才库（P0）

- 创建人才库分组并配置可见范围。
- 将候选人加入、移出或转移人才库。
- 支持标签、技能、适合职位、跟进人和下次联系日期。
- 支持全文和结构化检索。
- 支持从人才库创建新职位申请。
- 默认不将所有淘汰候选人永久入库，必须由 HR 明确选择。

### 12.10 报表（P0/P1）

P0：

- 职位候选人漏斗。
- 各阶段当前数量和平均停留时间。
- 简历解析成功率、规则通过率和 LLM 成功率。
- 面试数量、反馈完成率和反馈时效。

P1：

- 渠道质量、团队效率、招聘周期、人才库激活和自定义报表。

### 12.11 设置与审计（P0）

- 管理用户、角色、部门和职位可见范围。
- 管理筛选规则、面试评价模板和状态原因。
- 管理允许使用的 LLM Provider、模型和岗位范围。
- 查询登录、查看、下载、导出、状态变化、反馈、权限和配置审计。
- 管理候选人数据保留、删除和备份策略。

## 13. 现有筛选能力的融合

### 13.1 规则筛选

现有规则评分作为首期兼容基线：

- 必须条件命中占 75%。
- 加分项命中占 15%。
- 经验年限占 10%。
- 缺失任一必须条件时，现有兼容模式将总分上限限制为 59。

产品化后必须增加：

- 规则版本和生效时间。
- 规则来源与修改人。
- 命中、缺失和计算因子的结构化保存。
- 人工覆盖结论和原因。
- 后续支持职位级权重配置，但不作为 P0 上线阻塞项。

### 13.2 LLM 语义评估

- 默认关闭外部 LLM。
- 仅系统管理员可配置允许的 Provider 和 Base URL。
- API Key 保存在服务器密钥管理或环境变量中，不返回浏览器。
- 每次评分记录 Provider、模型、Prompt 版本、调用时间、结果和错误。
- 外发前按企业策略最小化字段，可选择脱敏姓名、联系方式和证件信息。
- LLM 返回评分、推荐结论、摘要、风险和建议面试问题。
- LLM 超时、429、格式错误或不可达时保留规则结果并允许重试。
- LLM 结果必须显示“AI 辅助、需人工复核”。
- LLM 不得直接执行淘汰、通过、发通知或删除候选人。

### 13.3 简历解析

P0 复用现有 PDF 文本层、DOCX 和文本文件解析能力，并增加结构化字段提取、质量状态和人工修正。

P1 增加：

- 扫描件和图片型 PDF OCR。
- DOCX 表格和复杂版式增强解析。
- CSV/XLSX 候选人清单导入。
- 解析模型或模板版本管理。

## 15. 隐私、安全与合规要求

### 15.1 数据分类

- 高敏个人信息：简历原文、联系方式、住址、证件、照片、薪资、面试评价和淘汰原因。
- 业务敏感信息：JD、HR 备注、候选人阶段和招聘统计。
- 系统敏感信息：API Key、OIDC 配置、会话、导出文件和审计日志。

### 15.2 必须控制

- 生产环境强制 HTTPS，不允许公网裸露业务端口。
- 简历附件使用私有存储和短期授权下载，不提供永久公网链接。
- 上传文件进行扩展名、MIME/magic、大小和数量校验。
- PDF/DOCX 解析任务设置超时、内存限制和隔离；生产环境接入恶意文件扫描。
- 下载、导出、查看简历和 LLM 调用必须审计。
- 日志不得记录简历正文、完整联系方式、会话、密码或 API Key。
- CSV/Excel 导出防止公式注入，导出范围必须经过权限过滤。
- 删除覆盖数据库、对象文件、搜索索引、临时缓存和可控备份窗口。
- 记录简历来源、用途、授权依据、人才库加入动作和保留期限。

### 15.3 数据保留

- 部署管理员必须在生产启用前确认保留策略。
- 默认情况下，候选人仅有终态申请、未加入人才库且连续 365 天无新活动时，进入删除候选队列。
- 明确加入人才库的候选人默认保留 730 天。
- 保留期限可按公司政策在 30 至 3650 天范围内配置。
- 存在进行中申请的候选人不因其他历史申请超期而进入删除队列。
- 候选人撤回或删除请求进入后台任务，并生成可追溯结果。
- 具体法域要求由部署企业的法务或合规负责人确认。

## 16. 非功能需求

### 16.1 性能与容量

- MVP 目标支持 50 名内部用户、10,000 名候选人。
- 单批最多 100 份简历，单文件最大 10MB，单批最大 500MB。
- 常用候选人列表查询在 10,000 名候选人规模下 P95 小于 1.5 秒。
- 上传、解析、OCR 和 LLM 评分使用后台任务，不占用长时间同步请求。
- 页面刷新、浏览器关闭和服务短暂重启后，任务进度可以恢复。

### 16.2 可用性与恢复

- 内部服务月可用性目标为 99.5%。
- 数据库每日自动备份，关键附件纳入对象存储备份。
- 恢复点目标 RPO 不高于 24 小时。
- 恢复时间目标 RTO 不高于 4 小时。
- 上线前完成一次候选人、附件、申请和反馈的一致性恢复演练。

### 16.3 浏览器与响应式

- 支持 Chrome、Edge、Safari 最近两个主要版本。
- 主要工作面向 1024px 及以上桌面浏览器。
- 窄屏支持查看待办、候选人概要和提交面试反馈。
- 批量筛选、复杂表格和管理设置不作为手机端主流程。

### 16.4 可访问性

- 所有状态必须有文字，不以颜色作为唯一表达。
- 表格支持键盘焦点、排序、筛选、分页和批量选择。
- 看板拖拽必须提供等价的菜单操作。
- 对话框具备焦点管理、Esc 关闭和明确的可访问名称。
- 长任务进度提供文本状态和当前数量。

### 16.5 可观测性

- 监控登录失败、API 错误率、任务积压、解析失败率、LLM 失败率和存储容量。
- 业务审计与系统日志分开存储。
- 异常告警不得包含候选人正文或敏感字段。
- 提供任务重试、失败原因和运维追踪 ID。

## 17. 推荐技术形态

采用单租户模块化单体，不在首期拆微服务：

```text
HR / 用人经理 / 面试官浏览器
                 |
             HTTPS / OIDC
                 |
               Nginx
                 |
       Web 应用与 REST API
          |              |
     PostgreSQL      后台任务 Worker
                         |
              简历解析 / 规则 / LLM / OCR
          |
   私有对象存储或受控 NAS
```

推荐基础组件：

- PostgreSQL：岗位、候选人、申请、面试、反馈、人才库和审计元数据。
- 私有对象存储：原始简历和受控导出文件。
- 后台任务队列：解析、筛选、OCR、LLM、通知和删除任务。
- Nginx 或同类入口：TLS、域名、请求限制和统一访问入口。
- Docker Compose：单服务器起步；后续根据容量再拆分数据库、对象存储和 Worker。

现有代码迁移边界：

- `resume_filter.py` 的文本提取迁入简历解析适配器。
- `score_resume` 迁入版本化规则筛选服务。
- `evaluate_with_llm` 迁入受控 LLM Gateway。
- 当前 CSV 字段保留为导出视图，不作为核心数据模型。
- 当前 `config.json` 中的岗位迁移为职位和 JD 版本。
- LLM 密钥不迁移到普通数据库配置或浏览器。

## 18. 验收标准

### 18.1 业务闭环

- HR 能创建职位并保存 JD 和筛选标准。
- HR 能导入多份简历，看到持久化进度和逐份结果。
- 每份成功简历形成候选人、简历和职位申请记录。
- HR 能按状态、评分、来源、关键词和时间筛选候选人。
- HR 能推进候选人状态并记录备注与淘汰原因。
- HR 能安排、改期和取消面试。
- 面试官能查看本人面试、提交反馈，不能读取无关候选人。
- HR 能根据反馈完成通过、淘汰、追加面试或入库决策。
- HR 能从人才库搜索候选人并创建新的职位申请。
- 所有关键操作在候选人时间线和审计日志中可追溯。

### 18.2 AI 与降级

- 同一规则版本、JD 版本和简历版本的规则计算结果稳定。
- 筛选结果记录所用规则、Prompt、模型和时间。
- LLM 未启用时产品仍能完成完整招聘流程。
- LLM 超时、429 或响应格式错误时规则结果不丢失。
- AI 结论不能自动改变职位申请状态。

### 18.3 权限和安全

- 未登录用户访问业务页面或 API 返回登录要求或 401/403。
- 直接访问无权限候选人 API 返回 403，不泄露摘要。
- 导出只包含当前用户有权查看的候选人和字段。
- API Key 不出现在页面响应、普通日志或导出中。
- 上传的伪装格式、超限文件和损坏文件被安全拒绝或隔离。
- 关键查看、下载、导出、修改和 LLM 调用均可审计。

### 18.4 质量和运维

- 自动化 E2E 跑通：创建职位 -> 导入简历 -> 筛选 -> 安排面试 -> 提交反馈 -> 决策 -> 加入人才库。
- 角色权限矩阵具备页面和 API 自动化验证。
- 100 份简历批量任务允许部分失败且可以重试。
- 数据库和附件备份恢复演练通过。
- 部署、升级、回滚、备份和故障排查步骤有可复现文档。

## 20. 分阶段路线

### 阶段 0：生产基础

- 新应用骨架、PostgreSQL、对象存储和后台任务。
- 登录、角色、部门、审计和 HTTPS。

- 人才库、标签、搜索和重新激活。

### 阶段 2：效率增强

- OCR 和复杂简历解析。

- 招聘漏斗、渠道和团队效率报表。

- 合法招聘渠道 API 集成。
- 候选人投递入口和内推。
- Offer、审批、背调和入职衔接。
- 更复杂的招聘自动化和人才运营。
- 多租户 SaaS 化。

## 21. 风险与应对

| 风险 | 应对 |
| --- | --- |
| 首期范围膨胀成完整 Moka 复制 | 以 MVP 闭环和非目标清单作为范围门槛 |

### 23.4 责任与任务归属

| 角色 | 负责 | 不负责 |
| --- | --- | --- |
| 产品负责人 | 目标、范围、优先级、业务规则、验收和方向选择 | 像素级页面和组件实现 |
| UX 设计 | 用户旅程、信息架构、任务流、线框、状态、原型和可用性测试 | 最终视觉 Token 和前端代码 |
| UI 设计 | 视觉方向、设计系统、组件、高保真、响应式标注和 Design QA | 修改业务规则或权限模型 |
| 前端工程 | 设计系统落地、页面交互、状态管理、可访问性和前端测试 | 擅自改变已确认任务流 |
| 后端工程 | 数据模型、权限、状态机、任务、接口和审计 | 以接口限制替代用户流程设计 |
| QA | 业务流程、权限、交互状态、响应式、可访问性和视觉验收 | 降低已确认验收门槛 |

UX 与 UI 任务分别记录在 UX/UI 规范中；每个业务模块的开发任务引用对应页面编号和任务编号。设计任务不是一个笼统的“做 UI”工单，也不混入后端功能工单。

### 23.5 设计与开发顺序

```text
PRD 确认
-> UX 信息架构与任务流
-> 低保真原型评审
-> UI 三套视觉方向选择
-> 设计系统与高保真页面
-> 技术实施计划
-> 前后端开发
-> Design QA
-> 可用性回归与发布
```

任何 P0 页面进入开发前，必须具备页面编号、角色权限、默认状态、加载/空/错误状态、主要操作、交互说明和验收条件。后端基础设施可以与 UX 设计并行，但依赖页面行为的数据接口不得在任务流确认前冻结。

## 24. 产品决策记录

已确认决策：
