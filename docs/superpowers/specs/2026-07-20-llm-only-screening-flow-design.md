# LLM 唯一评分与自动流转设计

## 状态

- 日期：2026-07-20
- 状态：产品设计已确认，等待实施计划
- 适用范围：服务端简历导入、LLM 筛选、候选人流转、人才库、用人经理待办和生产发布

## 背景

当前规则引擎只要发现任意一项必须条件没有被简历文本逐字命中，就会返回“需人工复核”，并将规则分限制在 59 分以内。该规则把宽松的第一轮初筛变成了硬条件终审，也无法识别同义表达、能力迁移和相近项目经验，造成大量 HR 人工复核并增加漏掉潜在候选人的风险。

初筛的职责只应是决定候选人是否值得进入用人经理评审，而不是代替用人经理作最终判断。产品决策是取消正常流程中的 HR 人工复核节点，使用 LLM 作为唯一评分来源，并对模型故障采用候选人友好的放行策略。

生产服务器同时承载 `aurora-tek.cn` 官网和 `hr.aurora-tek.cn` 招聘系统。当前 `beyondcandidate-proxy` 独占宿主机 443 端口，并包含后来添加的官网路由。官网路由尚未进入招聘系统仓库，现有发布脚本重建代理时存在覆盖官网配置的风险。

## 目标

1. 使用 LLM 对 JD 与简历进行多维综合评分，规则分不参与最终结论或流程流转。
2. LLM 总分达到 60 分时自动进入用人经理评审，不经过 HR 初筛。
3. LLM 总分低于 60 分时自动暂缓，并进入组织级人才库。
4. LLM 在有限重试后仍失败时，不因技术故障阻断候选人，自动进入用人经理评审。
5. 用人经理仍是是否沟通和是否安排面试的最终人工决策者。
6. 招聘系统发布不得覆盖、删除或中断官网的 HTTPS 路由。

## 非目标

- 本次不删除历史规则结果或重写历史筛选记录。
- 本次不使用规则分作为 LLM 不可用时的回退分数。
- 本次不改变恶意文件、格式不支持或完全无法读取文件的隔离策略。
- 本次不让 LLM 自动淘汰、安排面试或发送候选人外部通知。
- 本次不完成长期共享 edge gateway 的独立部署；只先为现有共享代理增加发布保护。

## 方案选择

采用“LLM 唯一评分 + 技术失败放行”方案。

未采用的方案：

- LLM 失败后等待服务恢复：会让候选人长期卡在系统中。
- LLM 失败后进入 HR 异常复核：重新引入了要取消的 HR 瓶颈。
- 所有候选人都进入用人经理评审：召回率最高，但失去初筛提效价值。

## LLM 评分合同

### 输入

只向已配置且允许使用的 Provider 发送现有脱敏流程处理后的：

- 当前岗位 JD；
- 当前简历解析文本。

现有规则命中、规则缺失和规则分不再进入新版本评分 Prompt，避免旧规则影响 LLM 的独立判断。姓名、手机号、邮箱、地址等信息继续遵循现有最小化和脱敏策略。

### 评分维度

LLM 必须返回以下维度分和证据，维度总分严格等于最终分：

| 维度 | 上限 | 判断范围 |
| --- | ---: | --- |
| 核心能力匹配 | 35 | 岗位核心技能、方法和工具的实际覆盖程度 |
| 相关经验与项目深度 | 25 | 相似项目、生产经验、问题复杂度和结果证据 |
| 职责与级别匹配 | 20 | 责任范围、独立性、协作和岗位级别匹配 |
| 能力迁移与成长潜力 | 10 | 相近能力、学习路径和可迁移经验 |
| 明确岗位约束 | 10 | 仅评价 JD 明确写出的非敏感约束，不推断受保护属性 |

每个维度都必须包含：

- `score`：不超过该维度上限的整数；
- `evidence`：仅引用简历和 JD 中存在的事实；
- `gaps`：缺失或需要用人经理确认的信息。

最终响应继续包含摘要、优势、风险和建议问题。服务端校验字段、类型、范围、维度合计和长度，拒绝不符合合同的响应并进入有限重试。

### 服务端派生结论

模型不再自行决定“需人工复核”。服务端只根据最终分和调用状态生成结论：

| 条件 | 系统结论 | 流转结果 |
| --- | --- | --- |
| 85–100 | 优先评审 | 自动进入用人经理评审 |
| 60–84 | 建议评审 | 自动进入用人经理评审 |
| 0–59 | 暂缓 | 自动进入人才库 |
| LLM 最终失败 | AI 评分不可用 | 自动进入用人经理评审 |

任何单项缺失只能影响对应维度分和风险证据，不允许作为一票否决条件。

## 状态与流转

### 正常流程

```text
简历导入
  -> 安全扫描与解析
  -> LLM 评分
      -> 60-100: application new -> review
      -> 0-59: application new -> deferred
      -> LLM 最终失败: application new -> review
  -> 用人经理评审
      -> 通过: contact / interview_pending
      -> 不通过: rejected
```

### `deferred` 状态

新增申请阶段 `deferred`，中文展示为“AI 初筛暂缓”。它与“已淘汰”不同：

- 该岗位当前不再主动推进；
- 候选人保留在组织人才库中；
- 不进入 HR 或用人经理必办列表；
- HR 可执行“转交用人经理”，将 `deferred -> review`；
- 所有自动和手动流转都记录阶段事件和审计日志。

`deferred` 仍视为该岗位的一条活动申请，防止重复导入产生多条并行申请。转交用人经理时复用原申请和原简历版本。

### LLM 失败放行

以下情况经过现有有限重试后执行放行：

- Provider 超时或不可达；
- 配额或限流；
- 响应格式或评分合同无效；
- Provider 或模型配置在任务执行时不可用。

放行时：

- LLM 分数保持为空，不伪造 60 分；
- 申请进入 `review`；
- 用人经理页面显示“AI 评分不可用，请直接查看简历”；
- 保存安全错误码、调用次数、Trace ID 和审计记录；
- 不向普通用户显示 API Key、Provider 原始响应或底层异常。

文件被安全扫描拒绝、文件格式不支持、文件不存在或无法创建候选人记录时不执行放行。这些情况继续留在技术失败列表，允许 HR 更换文件或重新上传。

## 原子性与幂等

一次 LLM 终态处理必须在同一数据库事务中完成：

1. 写入成功评价或最终失败状态；
2. 更新申请阶段；
3. 写入 `ApplicationStageEvent`；
4. 写入审计日志；
5. 创建用人经理待办事件，或创建人才库成员关系。

Worker 重试不得重复流转、重复通知或重复加入人才库。幂等键至少包含组织、筛选项、LLM 配置版本和目标动作。并发手动流转与自动流转使用现有候选人锁和申请版本校验，已经离开 `new` 的申请不得被旧任务覆盖。

## 用人经理待办与通知

进入 `review` 后创建持久化的“候选人待评审”任务，接收人为职位的 `hiring_owner_id`。工作台、通知中心和候选人详情使用同一任务来源，避免分别计算出不一致的数量。

- 待办内容只包含候选人、岗位、LLM 结论状态和跳转链接；
- 同一申请只存在一条活动待办；
- 用人经理提交通过或不通过后关闭待办；
- LLM 失败放行的待办带有“AI评分不可用”标记；
- 对于没有配置用人经理的历史岗位，待办分配给职位负责人并显示“岗位未配置用人经理”的配置警告；
- 新建或重新发布职位时，应明确要求配置用人经理。

本期只保证站内待办和通知中心一致，不要求同时发送飞书、邮件或短信。

## 暂缓人才库

每个组织使用一个招聘团队可见的系统人才库“AI 初筛暂缓”。首次需要时幂等创建，后续复用。

- 系统人才库首次创建时由组织招聘管理员负责；没有可用招聘管理员时，回退到首次触发创建的招聘人员，后续批次不得改写人才库负责人；
- 每条成员关系单独记录触发该批次的招聘人员，作为候选人的跟进负责人；
- 成员关系关联原申请和原职位；
- 原因记录为“LLM 初筛分低于 60”；
- 标签记录岗位名称、最终分和主要可迁移能力；
- 保留期限使用组织当前人才库保留策略；
- 已存在的成员关系只更新必要元数据，不重复创建；
- HR 手动转交用人经理后保留人才库关系作为来源记录。

- 待提交用人经理；

结果表改为：

任务汇总显示“已转交用人经理、已暂缓、AI评分不可用、文件处理失败”四类数量。评分完成后页面自动刷新为服务端真实阶段，不要求 HR 再选择和提交。

### 用人经理工作台

`review` 申请进入“待评审”列表。候选人行和通知均可直接打开脱敏简历、LLM 维度评分、证据、风险与建议问题。操作保持“通过评审”和“不通过”。

### 人才库

增加“AI 初筛暂缓”入口和筛选条件，展示原岗位、最终分、暂缓时间和主要缺口，并提供“转交用人经理”。该操作必须二次确认目标岗位仍开放且用人经理有效。

### 历史记录

历史规则结果和“需人工复核”记录继续只读展示，并标明“旧版规则结果”。新流程不得产生新的“需人工复核”结论。

## 数据与兼容性

实施需要：

- 扩展申请阶段约束，加入 `deferred`；
- 扩展 LLM 评价结构，保存维度评分与证据；
- 增加可持久化、可关闭且幂等的用人经理待办；
- 增加新的 Prompt 版本，不修改历史 Prompt；
- 为系统人才库增加可识别的用途或稳定标识，避免仅依赖可编辑名称；
- 保留现有规则表、规则结果和外键，当前版本只停止消费和展示规则结论。

迁移不得批量改变历史申请。只有新创建的筛选任务使用新流程；上线前仍处于 `new` 的旧申请保持原样，由运营人员决定是否重新评分。

## 生产 Nginx 发布保护

### 当前事实

- `beyondcandidate-proxy` 发布宿主机 443；
- `aurora-web` 通过共享 Docker 网络 `beyondcandidate_edge` 暴露 `aurora-web:3000`；
- 运行时 Nginx 同时包含 `hr.aurora-tek.cn`、`aurora-tek.cn` 和 `www.aurora-tek.cn`；
- 官网 server block 是线上追加配置，仓库模板当前不完整；
- `remote-release.sh` 继承 `.env` 和 Compose overlay，但没有继承上一个 release 的 Nginx 模板。

### 短期发布约束

招聘系统发布必须把上一个健康 release 的实际 Nginx 模板复制到新 release，再只替换招聘系统镜像版本。不得使用仓库中的单域名模板覆盖线上共享配置。

发布前必须校验继承配置：

- 精确包含 `server_name hr.aurora-tek.cn`；
- 精确包含 `server_name aurora-tek.cn`；
- 精确包含 `server_name www.aurora-tek.cn`；
- 官网路由仍指向 `http://aurora-web:3000`；
- 招聘 API 路由仍指向 `http://api:8000`；
- `aurora-web` 和 proxy 同时连接 `beyondcandidate_edge`；
- `nginx -t` 和 `docker compose config --quiet` 均通过。

发布后必须分别验证：

- `https://hr.aurora-tek.cn/health/ready` 返回 200；
- `https://hr.aurora-tek.cn/` 可完成招聘系统浏览器 smoke；
- `https://aurora-tek.cn/` 返回 2xx 或预期的 3xx；
- `https://www.aurora-tek.cn/` 返回 2xx 或预期的 3xx；
- 官网首页响应仍来自 `aurora-web`，不能仅凭状态码判断。

任一检查失败，发布脚本必须恢复上一个招聘 release 的服务和共享 Nginx 配置，并且不得停止、重建或删除 `aurora-web`。`docker compose -p beyondcandidate` 的清理操作不得使用 `--remove-orphans`。

### 长期方向

后续将 443、TLS 和多域名路由迁移到独立 edge gateway。届时招聘系统和官网只发布内部服务，任何单个应用的发布都不再拥有共享入口配置。该迁移另行设计，不阻塞本次筛选流程调整。

## 可观测性

至少新增或确认以下安全指标：

- LLM 评分成功、失败、重试和失败放行数量；
- 评分耗时和 token 用量；
- 自动进入用人经理评审数量；
- 自动暂缓数量；
- `deferred -> review` 手动恢复数量；
- 用人经理待办创建失败和重复抑制数量；
- 人才库写入失败数量；
- 三个生产域名发布前后 smoke 结果。

日志和审计不得包含简历正文、JD 全文、API Key 或 Provider 原始响应。

## 验收标准

1. 新导入简历的规则分不会影响最终分、结论或申请阶段。
2. LLM 返回 60 分时自动进入 `review`，59 分时自动进入 `deferred`。
3. 任意单项缺失不会直接产生“需人工复核”。
4. LLM 最终失败时分数为空，但申请自动进入 `review`，且用人经理看到明确标记。
5. `deferred` 候选人自动出现在“AI 初筛暂缓”人才库，不出现在 HR 或用人经理必办中。
6. HR 可将 `deferred` 候选人转交用人经理，重复点击不会产生重复申请或重复待办。
7. 新流程不再显示或产生“待HR审核、HR已审核、需人工复核、待提交用人经理”。
8. 历史规则结果保持可读且不会被迁移重写。
9. 部署招聘系统后，三个生产域名均通过各自 smoke，官网容器未被重建。
10. 发布失败回滚后，招聘系统和官网都保持上一个健康状态。

## 已决策事项

- LLM 是唯一评分来源。
- 60 分为自动进入用人经理评审的最低分。
- 低于 60 分自动暂缓并进入人才库。
- LLM 最终失败时放行到用人经理，不进入 HR 人工复核。
- 正常流程不再存在 HR 人工复核节点。
- 规则引擎先停止参与决策，历史数据暂不删除。
- 本期必须保护同服务器官网路由，禁止招聘发布覆盖共享 Nginx 配置。
