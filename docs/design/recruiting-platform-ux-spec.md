## 2. UX 目标

产品的 UX 目标不是展示功能数量，而是减少招聘过程中重复确认、上下文切换和遗漏。

核心目标：

1. HR 登录后能在一分钟内判断今天最需要处理的招聘事项。
2. HR 能在一个职位上下文中完成导入、筛选、推进、排面和查看结果。
3. 用人经理无需理解系统配置即可完成候选人判断。
4. 面试官无需学习完整 ATS 即可查看材料并在三分钟内提交反馈。
5. 任何长任务、失败和权限限制都给出明确状态和可执行下一步。
6. 人工结论、规则评分和 LLM 结论能够被区分和追溯。

## 3. UX 成功指标

| 核心任务 | 成功率目标 | 时间目标 | 允许错误 |
| --- | --- | --- | --- |
| 创建职位并保存筛选标准 | 90% | 5 分钟内 | 最多 1 次可恢复错误 |
| 为职位导入 20 份简历 | 95% | 3 分钟内发起 | 失败文件不影响其他文件 |
| 从筛选结果推进 5 名候选人 | 90% | 4 分钟内 | 最多 1 次误选并可撤销 |
| 安排一场面试 | 90% | 3 分钟内 | 时间冲突必须被发现 |
| 面试官提交反馈 | 90% | 3 分钟内 | 不得丢失已填写内容 |
| 从人才库重新激活候选人 | 85% | 3 分钟内 | 原历史不得被覆盖 |

总体目标：核心任务完成率不低于 85%，SUS 不低于 75，无阻断级可用性问题。

## 4. 角色与任务优先级

### 4.1 HR / 招聘专员

高频：

- 查看待办和新简历。
- 筛选、搜索和批量推进候选人。
- 添加备注、安排面试、催收反馈。

中频：

- 创建职位、调整 JD 和筛选规则。
- 归档候选人、加入人才库和导出结果。

低频：

- 合并候选人、处理数据删除请求。

UX 优先级：扫描效率、批量操作、上下文连续性、状态可见性。

### 4.2 招聘负责人

高频：

- 查看职位进展、团队积压和异常。
- 分配负责人、检查反馈时效。

中频：

- 查看报表、管理流程模板和人才库可见范围。

UX 优先级：跨职位概览、下钻分析、责任人和时间信息。

### 4.3 用人经理

高频：

- 查看被推荐候选人并给出建议。
- 查看面试结论和职位进度。

UX 优先级：低学习成本、业务证据集中、明确主操作。

### 4.4 面试官

高频：

- 查看本人面试与候选人资料。
- 提交结构化反馈。

UX 优先级：最少导航、最少必填、移动端可用、草稿不丢失。

### 4.5 系统管理员

中低频：

- 账号、角色、认证、LLM、审计和数据保留配置。

UX 优先级：风险提示、权限影响范围、二次确认、操作可追溯。

## 5. 用户旅程

| 阶段 | HR 行为 | 协作角色 | 当前痛点 | 设计机会 |
| --- | --- | --- | --- | --- |
| 建立职位 | 收集需求、写 JD、定义筛选条件 | 用人经理 | 公开 JD 与内部标准混在一起 | 分离公开 JD、内部标准和版本 |
| 收集简历 | 从授权来源下载或导出简历 | 无 | 文件分散、重复、格式不一 | 批量导入、来源标记、重复提示 |
| 初筛 | 阅读、比较、记录结论 | 用人经理 | 重复阅读、标准不统一 | 结构化摘要、规则/LLM 证据、批量动作 |
| 沟通推进 | 联系候选人并更新状态 | 用人经理 | 状态散落在聊天和表格 | 申请状态、备注和下一步待办 |
| 安排面试 | 协调人和时间 | 面试官、候选人 | 反复确认、改期容易遗漏 | 冲突检查、邀请文本、状态同步 |
| 面试与反馈 | 催反馈、汇总结论 | 面试官、用人经理 | 反馈格式不一、提交慢 | 短表单、草稿、待办提醒、汇总视图 |
| 决策与沉淀 | 通过、淘汰、入库 | 用人经理 | 历史记录丢失、人才难复用 | 终态原因、人才库关系、重新激活 |

### 6.3 导航规则

- 登录后默认进入与角色匹配的工作台。
- HR 的工作台优先展示新简历、待安排和待决策。
- 面试官的工作台优先展示今日面试和待反馈，不展示无关招聘管理入口。
- 用人经理默认只看到被授权职位和待判断候选人。
- 设置仅向有对应权限的角色显示，但后端仍执行独立授权。
- 页面标题、面包屑和返回路径必须保留对象上下文，不依赖浏览器后退。
- 全局搜索仅搜索当前用户有权访问的候选人和职位。

## 8. 核心任务流

### 8.1 F-01 创建职位

入口：`WB-01` 或 `JOB-01`。

主路径：

1. 点击“新建职位”。
2. 填写职位名称、部门、招聘人数、负责人和优先级。
3. 填写公开 JD。
4. 查看系统提取的必须条件和加分项并人工修正。
5. 选择流程模板和是否允许 LLM 辅助。
6. 保存草稿或发布为招聘中。

恢复路径：

- 离开未保存表单时提示保存草稿或放弃。
- JD 解析失败时保留原文本，允许手工录入条件。
- 无 LLM 使用权限时展示继承的组织设置，不展示密钥字段。

### 8.2 F-02 导入与筛选简历

入口：`JOB-03`。

主路径：

1. 点击“导入简历”，打开 `IMP-01`。
2. 选择合法来源、批次说明和文件。
3. 系统在提交前显示文件数量、总大小和不支持项。
4. 创建任务并进入 `SCR-01`。
5. 持续显示 `已完成/总数`、当前阶段和逐文件状态。
6. 完成后按人工状态、规则分和 LLM 分查看结果。
7. 批量推进、淘汰、加标签或加入人才库。

恢复路径：

- 失败文件单独展示原因和重试操作。
- 页面刷新后通过任务 ID 恢复进度。
- LLM 失败显示规则结果和“重试 LLM”，不显示整批失败。
- 重复候选人仅提示可能重复，人工确认后再合并。

### 8.3 F-03 候选人复核与推进

入口：`JOB-03`、`CAN-01` 或工作台待办。

主路径：

1. 打开 `CAN-02`。
2. 查看候选人摘要、当前申请、简历和筛选证据。
3. 添加备注或标签。
4. 选择下一状态并填写必要原因。
5. 状态更新写入时间线，页面保留当前位置。

恢复路径：

- 并发修改时展示新的服务端状态，并提供重新应用操作的入口。
- 非法状态跳转直接解释原因和允许的下一状态。
- 人工结论不删除规则或 LLM 证据。

### 8.4 F-04 安排或改期面试

入口：`CAN-02` 或 `INT-01`。

主路径：

1. 打开 `INT-02`，确认候选人和职位。
2. 选择轮次、方式、时区、时间、时长和面试官。
3. 系统显示已知冲突。
4. 填写地点或会议链接。
5. 确认邀请文本并保存。
6. 生成面试官待办、邀请文本和 `.ics` 文件。

恢复路径：

- 冲突时保留全部输入，允许修改时间或明确覆盖软冲突。
- 通知集成失败时显示失败渠道和重试，不删除面试安排。
- 改期创建变更记录，并使旧 `.ics` 失效。

### 8.5 F-05 面试反馈与招聘决策

入口：工作台待反馈或 `INT-01`。

主路径：

1. 面试官进入 `INT-03`。
2. 查看职位重点、候选人摘要、简历和建议问题。
3. 填写评价维度、优点、风险和结论。
4. 自动保存草稿。
5. 提交后进入只读确认状态。
6. HR 在 `CAN-02` 汇总反馈并完成决策。

恢复路径：

- 网络失败时保留本地草稿并提示重试。
- 已提交反馈的修改必须填写原因。
- 未完成必填项时聚焦第一个错误并保留其他内容。

### 8.6 F-06 人才库重新激活

入口：`TAL-02`。

主路径：

1. 搜索候选人并查看适合岗位和历史记录。
2. 选择“加入职位”。
3. 选择招聘中的目标职位和使用的简历版本。
4. 查看历史申请冲突或冷却期提示。
5. 创建新的职位申请并关联来源申请。

恢复路径：

- 同职位存在进行中申请时阻止重复创建并链接到现有申请。
- 历史申请不被覆盖。
- 无目标职位权限时不显示该职位。

## 9. 页面 UX 要求

### 9.1 WB-01 工作台

- 第一屏只展示可执行待办，不使用营销式欢迎区。
- 待办按紧急程度和到期时间排序，并明确对象、原因和主操作。
- 今日面试显示时间、候选人、职位、方式和反馈状态。
- 异常任务与普通待办分开，避免持续制造警报感。
- 无待办时展示今日面试或最近职位，不展示大面积空卡片。

### 9.2 JOB-01 / JOB-03 职位

- 职位列表默认展示招聘中，保留用户最近使用的筛选条件。
- 表格至少显示职位、部门、负责人、候选人数、待处理、面试中和更新时间。
- 职位详情默认打开候选人漏斗，但保留列表/看板视图偏好。
- 看板拖拽必须有等价状态菜单；批量操作优先在列表完成。
- JD、筛选规则和历史版本使用标签页，不与候选人主列表并排争抢空间。

### 9.3 SCR-01 筛选任务

- 顶部显示任务整体状态、当前阶段、数量、耗时和发起人。
- 逐文件状态支持“全部、处理中、成功、部分成功、失败”筛选。
- 失败原因使用人类可理解文案，技术追踪 ID 作为次要信息。
- 规则分与 LLM 分并列但视觉上区分来源。
- 批量推进前显示受影响人数和目标状态，完成后允许短时撤销非终态操作。

### 9.4 CAN-01 / CAN-02 候选人

- 候选人列表以比较为目标，默认使用表格而不是卡片瀑布流。
- 表格支持列配置、固定姓名列、排序、筛选、分页和批量选择。
- 详情页固定展示姓名、当前申请、当前状态、负责人和下一步。
- 简历、筛选、面试、人才库和活动通过标签页或时间线组织。
- 联系方式默认按权限脱敏，复制和下载有明确反馈。
- AI 结论始终标注来源和时间，不与人工结论混为一体。

### 9.5 INT-01 / INT-03 面试

- HR 默认看到面试列表，可切换周日历；面试官默认看到本人待办。
- 日历状态不能仅靠颜色区分。
- 反馈页保持单列，候选人材料使用可折叠摘要或侧栏。
- 自动保存状态可见但不抢占主操作。
- 提交反馈后明确告知下一步由谁处理。

### 9.6 TAL-01 / TAL-02 人才库

- 人才库不是候选人列表复制，必须突出分组、适合岗位、最近互动和保留期限。
- 搜索结果显示历史职位和最近结论，避免重复联系。
- “加入职位”是主要动作，“永久不再联系”和“黑名单”使用危险操作模式。
- 批量导出默认不作为主操作，并受权限与审计约束。

### 9.7 SET-01 至 SET-04 设置

- 设置按组织、招聘流程、AI 与数据治理分组。
- 高风险配置显示影响范围、当前状态和变更结果。
- LLM 配置测试结果在表单附近完整展示，长错误文案可换行。
- API Key 已保存时只显示掩码和替换操作，不回显原文。
- 删除、权限扩大、外部 Provider 启用和保留策略缩短需要二次确认。

## 10. 交互规范

### 10.1 搜索与筛选

- 页面主搜索框搜索当前对象，跨对象搜索放在全局顶部栏。
- 常用筛选器直接显示，低频筛选放入“更多筛选”。
- 已应用筛选显示为可逐项移除的条件摘要，并提供清空全部。
- 搜索和筛选条件写入 URL，便于刷新、返回和共享授权链接。
- 空搜索与无结果使用不同文案和操作。

### 10.2 表格与批量操作

- 表头固定，姓名或主对象列可固定。
- 选择后出现稳定的批量操作栏，不改变表格整体宽度。
- 高风险批量操作展示人数、对象范围和不可逆影响。
- 跨页全选必须明确是“本页”还是“全部筛选结果”。
- 状态、分数和日期支持排序；自由文本列默认不排序。

### 10.3 状态流转

- 主状态变更使用菜单或明确按钮，不只依赖拖拽。
- 状态菜单只展示当前允许的下一状态。
- 需要原因的状态在确认前收集原因。
- 操作完成后局部刷新，保持筛选、滚动位置和选中上下文。
- 终态不可直接重开；重新参与招聘创建新申请。

### 10.4 长任务

- 上传完成不等于解析完成，阶段文案必须区分。
- 进度显示数量、当前阶段和当前文件，不展示虚假百分比。
- 用户可离开任务页，工作台继续显示任务状态。
- 支持部分成功、失败项重试和已完成结果继续使用。

### 10.5 反馈与通知

- 成功反馈优先使用局部状态和短提示，不阻断后续操作。
- 需要用户处理的错误必须留在相关区域，不能只显示短暂 Toast。
- 删除、淘汰、覆盖冲突和权限变更使用模态确认。
- 系统通知必须指向具体对象和下一操作。

## 11. 页面状态矩阵

每个 P0 页面必须设计以下状态：

| 状态 | 设计要求 |
| --- | --- |
| 默认 | 展示真实可执行内容和明确主操作 |
| 首次空态 | 说明为什么为空，并提供创建或导入入口 |
| 筛选无结果 | 保留筛选条件，提供调整或清空操作 |
| 加载 | 保持稳定布局，不用整页旋转图遮挡 |
| 后台处理中 | 显示阶段、数量和可离开提示 |
| 部分成功 | 成功结果可继续使用，失败项可筛选和重试 |
| 错误 | 说明对象、原因、影响和下一步 |
| 无权限 | 不泄露对象摘要，提供返回安全页面的操作 |
| 禁用 | 说明为何不可用以及满足条件 |
| 未保存 | 离开前提示保存草稿或放弃 |
| 并发冲突 | 展示服务器最新状态，允许刷新或重新应用 |
| 删除/归档 | 使用二次确认并说明可恢复性和保留策略 |

## 12. 内容与文案规则

- 使用招聘业务语言，不暴露内部数据表名或任务队列术语。
- 状态使用一致词汇，例如统一使用“待复核”，不混用“待审核”。
- 错误文案结构为“发生了什么 + 影响什么 + 可以怎么做”。
- AI 内容使用“AI 辅助评分”“AI 风险提示”，不使用“AI 判定”。
- 时间同时显示绝对时间；近期活动可补充“3 小时前”等相对时间。
- 危险动作使用具体动词，例如“淘汰 12 名候选人”，不使用泛化“确认”。
- 不在页面中放置说明产品功能或视觉设计的宣传性文案。

## 13. 权限体验

- 导航和操作按权限减少噪声，但服务端权限错误仍有独立状态。
- 面试官只能从本人待办进入候选人，不提供跨候选人搜索。
- 用人经理看到筛选证据和面试结论，但联系方式默认脱敏。
- 系统管理员管理系统配置，不因管理员身份自动获得简历数据权限。
- 无权限页面不显示姓名、职位申请摘要或其他敏感信息。
- 导出和下载显示审计提示，但不使用过度恐吓文案。

## 14. 响应式策略

| 视口 | 策略 |
| --- | --- |
| 1440px 及以上 | 固定侧栏、完整表格、详情双栏或三栏 |
| 1024–1439px | 可收起侧栏、隐藏低优先级列、详情双栏 |
| 840–1023px | 抽屉导航、单栏详情、批量工具横向滚动 |
| 840px 以下 | 面试官待办、候选人摘要和反馈单列；不支持复杂批量筛选 |

响应式不能依靠缩小字体解决空间问题。低优先级信息折叠到详情或菜单，主操作始终可见且不与文本重叠。

## 15. 可访问性要求

- 目标达到 WCAG 2.2 AA。
- 正文和控件文字对比度不低于 4.5:1，大字号不低于 3:1。
- 所有交互元素有可见焦点，键盘顺序与视觉顺序一致。
- 表格排序、筛选和选择状态可被辅助技术读取。
- 状态不能只使用颜色表达，必须有文字或图标加文字。
- 看板拖拽提供键盘和菜单替代操作。
- 对话框打开后管理焦点，Esc 可关闭非阻断对话框。
- 自动保存、长任务和错误状态使用适当的可读状态提示。
- 触控目标最小 40×40px；紧凑表格中的图标操作仍保持可点击范围。

## 16. 可用性测试计划

### 16.1 参与者

- 4 名日常筛选简历的 HR。
- 2 名用人经理。
- 2 名面试官。
- 至少 1 名参与者使用 1024px 笔记本视口。

### 16.2 测试任务

1. 创建 AI 工程师职位并确认筛选条件。
2. 导入一批包含一个失败文件和一个重复候选人的简历。
3. 找出三名合适候选人并推进到待沟通。
4. 为其中一名候选人安排并改期面试。
5. 作为面试官提交反馈。
6. 将未录用候选人加入人才库，并从人才库重新加入新职位。

### 16.3 记录指标

- 任务完成率。
- 完成时间。
- 错误和回退次数。
- 需要主持人提示的次数。
- 对评分可信度和权限边界的理解。
- SUS 分数和开放反馈。

### 16.4 严重级别

| 级别 | 定义 | 发布处理 |
| --- | --- | --- |
| S0 阻断 | 核心任务无法完成或数据丢失 | 必须修复并复测 |
| S1 严重 | 高概率误操作、越权或错误决策 | 必须修复并复测 |
| S2 中等 | 明显降低效率但有可发现绕行路径 | 进入当前迭代 |
| S3 轻微 | 文案、对齐或低频可发现性问题 | 排入后续优化 |

## 17. UX 交付物与门禁

### 门禁 UX-1：信息架构

交付：站点结构、角色导航、页面编号和对象关系。

通过标准：产品、UX 和架构确认页面归属、权限和核心对象一致。

### 门禁 UX-2：低保真流程

交付：F-01 至 F-06 的低保真线框图，覆盖主路径和至少一个错误状态。

通过标准：HR 代表完成任务走查，未发现缺失页面或断裂状态。

### 门禁 UX-3：中保真原型

交付：可点击原型、真实示例数据、桌面主流程和面试官窄屏反馈。

通过标准：8 名目标用户完成测试，核心任务成功率不低于 85%，S0/S1 为零。

### 门禁 UX-4：工程确认

交付：页面状态矩阵、交互说明、权限差异和数据依赖。

通过标准：前端、后端和 QA 能据此拆任务和编写验收用例，无关键行为歧义。
